# План разработки backend на Next.js для pstock

## Цели и контекст
- Реализовать backend внутри Next.js (App Router) под те же REST‑эндпоинты, что уже используются веб‑приложением и телеграм‑ботом.
- Сохранить бизнес‑логику из текущих `services` и схему БД PostgreSQL из `migrations/001_create_tables.sql`.
- Обеспечить совместимость с фронтендом (в т.ч. Telegram Web App), который обращается к `window.location.origin + '/api'`.
- Сохранить интеграции: генерация Excel отчёта, отправка уведомлений в канал, правила уникальности пересчёта/привоза.

## Технологии
- Next.js 14 (App Router, Node.js runtime для API‑роутов).
- PostgreSQL (`pg`, пул подключений, `DATABASE_URL`).
- Node‑утилиты для Excel (`xlsx`) и Telegram (HTTP Bot API или существующий сервис).
- Логирование (`winston`) и централизованная обработка ошибок.

## Архитектура
- Разместить API‑роуты в `backend/app/api/*/route.ts` (или `.js`) со строгими контрактами.
- Вынести доступ к БД и доменную логику в раздел `backend/lib/*`:
  - `lib/db.ts` — пул к БД, конфиг из `.env`.
  - `lib/repositories/*` — SQL‑запросы (минимум по модулям: addresses, categories, items, stock, replenish, reports, users, rights).
  - `lib/services/*` — бизнес‑операции, максимально повторяя поведение из `services/*.js`.
- Интеграция с Telegram:
  - В проде предпочтительно оставить бота отдельным процессом (как сейчас) и взаимодействовать через общую БД и канал уведомлений.
  - Для уведомлений из API использовать HTTP Bot API (отправка сообщений/документов по токену).

## Схема БД (перенос без изменений)
- Таблицы: `rights`, `users`, `address`, `category`, `item`, `stockreport`, `stock`, `replenish`, `replenishstock`.
- Уникальности и индексы — как в `migrations/001_create_tables.sql`.
- Начальные данные: права, адреса, категории.

## Эндпоинты (Next.js Route Handlers)
- `GET /api/addresses` — список адресов.
- `GET /api/categories/:addressId` — категории по адресу.
- `GET /api/items/:addressId/:categoryId` — позиции для пересчёта.
- `GET /api/items/search/:addressId/:categoryId?query=...` — поиск позиций (минимум 3 символа).
- `POST /api/stock/create-report` — создать/обновить отчёт пересчёта за дату; контракт: `{ addressId, categoryId, items: [{ itemId?|0, name?, qty }], date? }`.
- `PUT /api/stock/update-report/:reportId` — обновить записи отчёта.
- `GET /api/stock/report/:reportId` — получить отчёт и его позиции.
- `POST /api/stock/generate-order/:reportId` — рассчитать недостающее (по `expected`), отправить в канал.
- `GET /api/stock/order/:reportId` — вернуть сформированный заказ.
- `POST /api/replenish/create` — создать привоз; контракт: `{ addressId, items: [{ itemId, qty }], date? }` с блокировкой дублей по дате и набору позиций.
- `PUT /api/replenish/update/:replenishId` — заменить позиции привоза.
- `GET /api/replenish/:replenishId` — детали привоза с позициями.
- `GET /api/replenish/copy-text/:replenishId` — текст для копирования.
- `POST /api/reports/generate` — Excel за выбранный месяц `YYYY-MM` (лист «Привозы», лист «Пересчёты»), ответ — бинарный файл.

## Контракты данных и правила
- Пересчёт:
  - В день для пары `(address, category)` — один `stockreport`; повторное сохранение перезаписывает записи `stock` отчёта.
  - Новые позиции (без `itemId`) добавляются в `item` с `address_id=0` (не участвуют в будущем автозаполнении).
  - После первого сохранения — вернуть `reportId`, разблокировать «Сформировать заказ», «Изменить».
- Заказ:
  - Позиции, у которых `qty >= expected`, не включать.
  - Отправка в канал при наличии `ORDER_CHANNEL_ID`.
- Привоз:
  - Уникальность по дате, адресу и набору позиций; повторное сохранение того же — ошибка.
  - После сохранения вернуть `replenishId` и текст для копирования.

## Аутентификация и доступ
- Веб‑приложение: изначально без сессий (как сейчас), с возможной дальнейшей интеграцией Telegram Web App `initData` для привязки пользователя.
- Бот остаётся источником аутентификации (логин/пароль) и прав (`rights_id`).
- При необходимости — добавить простую middleware‑проверку пользователя по токену/сеансу в API.

## Обработка ошибок и логирование
- В API‑роутах: try/catch, ответы с описанием и кодом.
- Отправка критических ошибок в канал `ERROR_CHANNEL_ID`.
- Логи — через `winston`, формат: уровень, метка, трассировка.

## Миграции и окружение
- Использовать существующие миграции: `npm run migrate` из корня.
- Переменные окружения:
  - `DATABASE_URL`, `NODE_ENV`, `ORDER_CHANNEL_ID`, `ERROR_CHANNEL_ID`, `TELEGRAM_BOT_TOKEN`, `WEB_APP_URL`.

## Пошаговый план реализации
1) Инициализировать модуль БД `backend/lib/db.ts` (пул `pg`).
2) Перенести SQL из `routes/api/*` в `lib/repositories/*` и бизнес‑логику из `services/*` в `lib/services/*`.
3) Создать `backend/app/api/*/route.ts` с теми же путями и контрактами.
4) Реализовать генерацию Excel с возвратом бинарного ответа, очистка временных файлов.
5) Добавить отправку сообщений/документов в канал через Bot API.
6) Обновить фронтенд, при необходимости — `API_BASE_URL = window.location.origin + '/api'` уже совместим.
7) Прогнать миграции, проверить CRUD:
   - Адреса/категории/поиск/позиции.
   - Пересчёт: create/update/get, generate‑order, order.
   - Привоз: create/update/get, copy‑text.
   - Отчёт: generate (excel).
8) Нагрузочные проверки (поиск ≥3 символов, большие списки, сортировка, лимиты).
9) Добавить базовые e2e/интеграционные тесты для API.

## Проверки совместимости
- Веб‑приложение использует эндпоинты из `public/js/api.js` — контракты API в `backend/app/api/` должны с ними совпадать.
- Бот продолжает работать с общей БД и каналами, `WEB_APP_URL` указывает на Next.
- HTTPS обязателен для кнопки Web App в Telegram; локально — падать обратно на обычную URL‑кнопку.

## Риски и меры
- Долгие операции (Excel) — потоковый ответ, тайм‑ауты и информирование пользователя о ожидании.
- Ограничения Vercel/FaaS: для бота — отдельный процесс; для API — Node runtime, без долгих опросов.
- Консистентность данных: транзакции при create/update, уникальные индексы, идемпотентность.

## Итог
- После переноса API в Next.js сохраняется совместимость, коды сервисов остаются переиспользуемыми, а деплой упрощается за счёт единого приложения.
